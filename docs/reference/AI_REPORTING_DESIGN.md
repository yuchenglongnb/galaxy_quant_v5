# AI 报告与方法论沉淀设计

## 问题背景

当前竞价报告中的趋势描述主要由硬编码规则生成。例如：

```text
上证趋势: 连涨+阳线
```

这个标签来自近两日涨跌幅、昨日实体涨跌、竞价涨幅和量比等少量条件组合。它的问题是：

- 只能表达离散标签，不能表达 K 线结构的复杂语境。
- 容易把“昨日上涨”简化成“趋势向上”，忽略当日高低点、影线、均线、位置、量能和压力位。
- 不能根据历史案例持续进化。
- 报告结论缺少证据链，用户难以判断“为什么这么说”。

截图中的上证指数更像是：

- 高位震荡后的回落/弱修复结构。
- 上方均线或前高附近仍有压力。
- 缩量环境下，反弹持续性需要验证。
- 当日低开后未形成强修复，不能简单表述为“连涨+阳线”。

因此，AI 不应替代底层行情计算，而应替代“语义归纳、场景解释、证据组织、方法论复用”这些硬编码最薄弱的环节。

## 总体原则

```text
确定性计算负责“事实”
AI 负责“解释”
RAG 负责“方法论和案例召回”
人工反馈负责“进化”
```

不要让 AI 直接读原始 CSV 后自由发挥。更稳妥的路径是：

1. 程序先计算结构化特征。
2. RAG 检索方法论和历史案例。
3. LLM 基于事实和检索材料输出标准 JSON。
4. 程序校验 JSON、落日志、生成报告。
5. 人工反馈沉淀回方法论库。

## 适合引入 AI 的环节

### 1. 指数趋势研判

当前位置：

- `analyzers/auction.py::_analyze_trend_bias`
- `runners/auction.py::_print_market_environment`
- `runners/auction.py::_print_indices_monitor`

建议替代内容：

- `趋势` 字段。
- `研判` 字段。
- `警示` 字段。
- 市场环境说明。

不建议替代内容：

- `auction_pct`
- `close_pct`
- `body_pct`
- `vol_ratio`
- `OAR`

这些仍由代码计算。

### 2. ETF/行业信号解释

当前 CP/SA 数值可以保留，但评语模板可以 AI 化。

适合 AI 替代：

- “为什么这是诱多/反核/趋势延续”
- “需要观察哪些确认条件”
- “与历史案例相似在哪里”

不适合 AI 替代：

- CP/SA 分数。
- 排名。
- 阈值触发。
- 胜率统计。

### 3. 盘后复盘总结

适合 AI 生成：

- 今日主线结构总结。
- 强弱扩散路径。
- 次日观察清单。
- 风险和失效条件。

底层候选池仍由规则筛选，AI 负责归纳。

## AI 输入特征设计

以“上证指数趋势研判”为例，AI 输入不应是截图，而应是结构化事实：

```json
{
  "target": "上证指数",
  "date": 20260526,
  "bars": [
    {"date": 20260520, "open": 4090.12, "high": 4217.07, "low": 4067.75, "close": 4160.31, "pct": 1.2, "amount": 1.4e12},
    {"date": 20260521, "open": 4165.0, "high": 4180.0, "low": 4100.0, "close": 4120.0, "pct": -0.8, "amount": 1.2e12}
  ],
  "today": {
    "open": 4137.32,
    "high": 4140.50,
    "low": 4104.46,
    "close": 4119.91,
    "pct": -0.79,
    "body_pct": -0.42,
    "upper_shadow_pct": 0.08,
    "lower_shadow_pct": 0.37,
    "vol_ratio": 0.69
  },
  "moving_averages": {
    "ma5": 4124.97,
    "ma10": 4145.70,
    "ma13": 4161.63,
    "ma30": 4113.61
  },
  "position": {
    "pos_5d": 35,
    "distance_to_ma5_pct": -0.12,
    "distance_to_ma13_pct": -1.00,
    "near_pressure": ["ma5", "ma10", "ma13"],
    "near_support": ["ma30", "previous_low"]
  },
  "market_context": {
    "oar": 0.69,
    "volume_state": "缩量"
  }
}
```

需要新增的特征：

| 特征 | 用途 |
| --- | --- |
| 上下影线比例 | 区分承压、探底、十字星 |
| 实体方向和实体强度 | 区分真阳/假阳/弱阴/大阴 |
| MA5/MA10/MA13/MA30 距离 | 描述均线压力和支撑 |
| 近 N 日高低位置 | 判断高位/中位/低位 |
| 近 N 日结构 | 连涨、连跌、反弹、回落、箱体 |
| 量价关系 | 放量确认、缩量反弹、缩量下跌 |
| 跳空缺口 | 竞价风险和情绪变化 |
| 前高/前低距离 | 压力位和支撑位 |

## RAG 知识库设计

目录建议：

```text
knowledge/
├── methodology/
│   ├── index_trend.md
│   ├── auction_cp_sa.md
│   ├── volume_price.md
│   ├── moving_average.md
│   └── failure_patterns.md
├── cases/
│   ├── index/
│   │   └── 20260526_000001_SH.md
│   ├── etf/
│   └── industry/
└── feedback/
    └── corrections.md
```

### methodology

沉淀稳定方法论，例如：

- 缩量反弹如何判断。
- 高位低开如何区分洗盘和转弱。
- MA5/MA13/MA30 的压力和支撑解释。
- CP 高但日内走强时如何避免误判。
- SA 高但实体未修复时如何降级为弱势承接。

### cases

沉淀历史案例，每个案例包含：

```text
日期:
标的:
当时数据:
原报告结论:
人工修正结论:
后续走势:
方法论沉淀:
```

### feedback

记录用户纠错：

```text
2026-05-26 上证指数
原输出: 连涨+阳线
用户反馈: 与 K 线明显不符，实际是缩量回落/均线承压
修正标签: 缩量承压回落
新增规则: 若当日收跌、实体为阴、收盘低于 MA5 且 OAR<0.8，不应输出“连涨+阳线”
```

## LLM 输出协议

AI 输出必须是可校验 JSON，而不是自由文本：

```json
{
  "label": "缩量承压回落",
  "bias": "偏弱震荡",
  "confidence": 0.78,
  "evidence": [
    "当日收跌 -0.79%，实体为阴",
    "OAR=0.69，量能缩至 0.8 以下",
    "收盘低于 MA5，MA10/MA13 在上方形成压力"
  ],
  "watch_points": [
    "能否重新站回 MA5",
    "4104 附近低点是否继续失守",
    "量能是否放大修复"
  ],
  "invalid_if": [
    "次日放量收复 MA5 和 MA10",
    "金融或权重板块带量上行"
  ],
  "report_text": "上证处于缩量承压回落状态，短线仍在均线压力下震荡，需观察能否重新站回 MA5。"
}
```

程序侧校验：

- `label` 必须在允许标签集中，或进入待审核新标签。
- `evidence` 必须引用输入数据中的字段。
- `confidence < 0.6` 时报告标注“低置信度”。
- 若 AI 输出和硬规则明显冲突，保留硬规则数值，报告显示“AI 研判待确认”。

## 标签体系建议

指数趋势标签不要只用“连涨/连跌/阳线/阴线”，应扩展为结构标签：

| 标签 | 表征 |
| --- | --- |
| 放量突破 | 带量站上关键均线/前高 |
| 缩量反弹 | 涨但量不足，延续性待验 |
| 缩量承压回落 | 缩量下跌且上方均线压制 |
| 高位震荡 | 近 N 日高位但无方向突破 |
| 均线压制 | 收盘低于短中期均线 |
| 回踩支撑 | 下探支撑后收回 |
| 破位转弱 | 跌破关键均线或前低 |
| 探底修复 | 下影较长且收回重要位置 |
| 冲高回落 | 上影明显，追高承压 |
| 低位反弹 | 低位放量或实体修复 |

## 落地阶段

### 阶段一：AI 只做旁路点评

不影响原报告排序和信号，只在报告中新增：

```text
AI趋势解读:
AI证据:
AI观察点:
```

优点：

- 风险小。
- 可以和硬编码结果对照。
- 便于收集纠错样本。

### 阶段二：AI 替代趋势描述

将 `上证趋势`、`趋势研判` 的文本改为 AI 输出。

保留硬规则：

- 数值计算。
- CP/SA 触发。
- 排名。
- 胜率统计。

### 阶段三：RAG 加入方法论和案例

每次生成报告时检索：

- 方法论片段。
- 相似历史案例。
- 用户近期纠错。

AI 输出同时给出：

- 当前判断。
- 相似案例。
- 与历史案例的差异。

### 阶段四：反馈闭环

报告中支持人工修正：

```text
原标签: 连涨+阳线
修正标签: 缩量承压回落
原因: 收阴、缩量、低于 MA5、上方 MA13 压制
```

修正进入 `knowledge/feedback/`，定期归并到 `knowledge/methodology/`。

## 方法论 Skill 沉淀

可以把“AI 如何解释市场结构”做成项目内 skill。

建议路径：

```text
skills/
└── market_report_interpreter/
    ├── SKILL.md
    ├── label_taxonomy.md
    ├── prompt.md
    └── examples.md
```

`SKILL.md` 应包含：

- 输入字段说明。
- 不能改动的事实字段。
- 标签选择原则。
- 证据输出格式。
- 低置信度处理。
- 用户纠错后如何更新案例。

这个 skill 不等同于模型提示词，而是项目的方法论协议。它让后续无论换模型、换 RAG 引擎，解释口径都能持续积累。

## 推荐技术架构

```text
DataManager / DataProcessor
  -> FeatureBuilder 生成结构化行情特征
  -> MethodologyRetriever 检索方法论和案例
  -> AIInterpreter 输出 JSON 研判
  -> Validator 校验证据和标签
  -> ReportRenderer 渲染报告
  -> FeedbackCollector 收集人工纠错
```

新增模块建议：

```text
ai/
├── feature_builder.py
├── retriever.py
├── interpreter.py
├── schemas.py
└── feedback.py
```

## 第一版最小实现

先做一个最小闭环：

1. 新增 `FeatureBuilder.build_index_context()`。
2. 为四大指数生成 10 日 OHLCV、均线、位置、影线、量能结构。
3. 新增 `docs/knowledge/methodology/index_trend.md`。
4. 新增 `AIInterpreter.interpret_index_trend(context)`。
5. 在报告中并排显示：

```text
硬规则趋势: 连涨+阳线
AI趋势: 缩量承压回落
证据: 收跌、缩量、低于MA5、MA13压制
```

等积累 20-50 个修正案例后，再替代硬规则文本。

## 风控边界

- AI 不能编造行情数据。
- AI 不能直接给买卖指令。
- AI 不能覆盖 CP/SA 数值。
- AI 的每个结论必须有证据字段。
- AI 输出失败时回退到硬规则报告。
- 每次 AI 版本或方法论更新要记录在报告日志中。
